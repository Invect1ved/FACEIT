# FACEIT Scout & Match Tracker

## Описание проекта

Учебное Android-приложение для учёта профилей игроков (никнейм, эло, уровень, ссылки и комментарии) и результатов матчей в духе FACEIT: главный экран с вкладками «Обзор», «История» и «Команда», профиль игрока, формы добавления и редактирования, локальное хранение в **Room**. Интерфейс (кнопки, диалоги, пустые состояния) на **русском языке**. Проект не связан с FACEIT Ltd.

## Функциональные возможности

- **Команда:** список всех игроков из базы на вкладке главного экрана; открытие профиля по нажатию; удаление свайпом влево или через диалог по иконке корзины; кнопка «+» для добавления нового игрока.
- **Обзор и история:** карточка «моё эло» (профиль с минимальным id в базе), карты, синтетическая кривая эло по матчам, агрегированная статистика; история матчей с дельтой эло.
- **Профиль игрока:** данные профиля, список матчей, добавление и **редактирование** матча, удаление матча с подтверждением, переход к редактированию профиля.
- **Формы:** создание и правка игрока; создание и правка матча (карта, победа/поражение, K/D, дата и время).

## Используемые технологии

| Область | Технологии |
|--------|------------|

| Сборка | **Gradle** (Kotlin DSL), **AGP 9** |

## Инструкции по запуску

1. Установите **Android Studio** (рекомендуется актуальная стабильная ветка, например Ladybug или новее).
2. Используйте **JDK 17+** (как правило, встроенный JDK Android Studio).
3. Клонируйте репозиторий и откройте каталог проекта в Android Studio.
4. Выполните **File → Sync Project with Gradle Files**.
5. Подключите устройство или эмулятор с **API 24+**, запустите конфигурацию **Run** для модуля `app`.
6. Сборка из командной строки (Windows): `gradlew.bat assembleDebug` — готовый APK: `app/build/outputs/apk/debug/`.

## Работа с Git и GitHub (для отчёта по лабораторной)

### 1. Создание репозитория на GitHub и привязка к проекту

1. На GitHub: **New repository** → имя (например `FACEIT-scout`), без README, если он уже есть локально.
2. Создайте **пустой** репозиторий на GitHub с именем **`FACEIT`** (без README, если проект уже есть локально):  
   https://github.com/new → Owner **Invect1ved**, Repository name **FACEIT** → Create repository.

3. В каталоге проекта (если Git ещё не настроен):

```bash
git init -b main
git remote add origin https://github.com/Invect1ved/FACEIT.git
git add .
git commit -m "feat: начальная версия приложения FACEIT Scout"
git push -u origin main
```

Удалённый репозиторий для этого проекта: **https://github.com/Invect1ved/FACEIT**  
Для первого `push` по HTTPS GitHub запросит авторизацию (**Personal Access Token** с правом `repo` или вход через браузер в новых версиях Git Credential Manager). По SSH: `git@github.com:Invect1ved/FACEIT.git`.

### 2. Осмысленные коммиты

Сообщения должны отражать **законченный** смысл изменения, на русском или на английском в едином стиле, например:

- `feat(ui): вкладка «Команда» со свайп-удалением игроков`
- `feat(data): слой репозитория над PlayerDao и MatchDao`
- `fix: корректные границы эло при симуляции кривой на дашборде`

Избегайте неинформативных сообщений вроде `fix`, `upd`, `123`.

### 3. Ветвление и стабильная `main`

- Ветка **`main`**: только собираемое и проверенное состояние.
- Новые возможности: ветки вида `feature/имя-задачи` от `main`:

```bash
git checkout -b feature/match-edit-screen
# ... правки, коммиты ...
git checkout main
git merge feature/match-edit-screen
```

При работе в команде часто используют **Pull Request** на GitHub вместо прямого merge в локальной копии.

### 4. Слияние (merge) и конфликты

- Обычное слияние: `git merge <ветка>` (или merge PR на сайте GitHub).
- Если Git сообщает о **конфликте**: откройте указанные файлы, найдите маркеры `<<<<<<<`, `=======`, `>>>>>>>`, вручную оставьте нужный код, затем `git add <файлы>` и `git commit` (или завершите merge в UI Android Studio / GitHub).

### 5. Регулярный `git push`

После логичных порций работы:

```bash
git push origin main
# или
git push origin feature/ваша-ветка
```

Так на GitHub отражаются этапы разработки для преподавателя.

### 6. Что приложить к отчёту (по методичке)

- **Ссылка** на публичный репозиторий GitHub.
- **Скриншот** страницы **Commits** (или `git log --oneline --graph --all` в терминале).
- **Скриншот** веток: GitHub → **Branches**, или вывод `git branch -a`.
- Краткий **текстовый конспект этапов**: что делали в какой ветке, когда сливали в `main`, пример одного merge (и при необходимости — как решали конфликт).

### Пример именования веток под этапы курса

| Ветка | Назначение |
|--------|------------|
| `main` | Стабильная сдаваемая версия |
| `feature/data-room` | Модель и Room |
| `feature/ui-dashboard` | Главный экран и вкладки |
| `feature/match-crud` | Матчи: форма и редактирование |

## Замечание по AGP 9

В `gradle.properties` может быть задано `android.disallowKotlinSourceSets=false` для корректной работы KSP (Room/Hilt) с текущей цепочкой AGP. При обновлении инструментов имеет смысл перепроверить этот флаг в документации Android Gradle Plugin.
